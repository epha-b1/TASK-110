# Task Prompt

Implement a Hospitality Operations Intelligence & Group Itinerary API for offline hotel groups and property management teams to coordinate guest movement plans while producing operational reporting from locally stored data. The backend, built with Express using Sequelize over MySQL, exposes Authentication and Account APIs that support only local username + password login (minimum 10 characters, at least 1 number and 1 symbol), role-based authorization, profile management (legal name, US mailing address, tax invoice title, preferred currency display), and self-service account deletion plus a data export entry point that packages the user’s profile, activity logs, and uploaded files into a locally generated archive. Core collaboration is provided through Itinerary Group APIs that support creating groups, inviting members by pre-shared join code, and maintaining structured itinerary items such as meetup time (MM/DD/YYYY, 12-hour time), meetup location text, route checkpoints (ordered, max 30), and notes (max 2,000 chars). Groups can configure required member data fields (e.g., vehicle make/model/license plate, emergency contact name and US phone format) with per-field required/optional flags; missing required fields block check-in to a meetup. File/attachment APIs support image/document upload up to 10 MB per file with MIME-type allowlists and SHA-256 fingerprinting for deduplication; access control is enforced so only group members can read, while only owners/admins can delete. Change notifications are implemented as in-system event records (no email/SMS/push) that members can query by cursor, including what changed, who changed it, and when, with idempotency keys to prevent duplicate updates during retries.

The data query and reporting center provides Reporting and Export APIs that support configurable filters and aggregations over reservations and room inventory, including occupancy rate, ADR, RevPAR, revenue mix by channel, and rollups by day/week/month, room type, and channel. Roles are Hotel Administrator (full access), Manager (property-scoped access), Analyst (read-only reporting), and Member (itinerary-only). Permission rules constrain visibility to assigned properties and prevent export of PII unless explicitly granted; exported CSV/Excel files are generated locally and logged with who/when/filters used. Data import APIs accept template-based Excel for staffing and evaluation datasets with strict column validation, duplicate merge rules (match on employee_id + effective_date), and an error receipt artifact listing row number, field, and reason; reports include position distribution, staffing gaps, evaluation results, and rewards/penalties with signed-off-by metadata. Face enrollment APIs support guided capture metadata (multi-angle required: left/front/right) and local liveness checks based on blink timing, motion consistency, and texture/reflection heuristics; only encrypted face templates are stored (AES-256 at rest) with versioning, update, and deactivation, and raw images are optional with default retention of 24 hours then hard-delete. Data quality and observability requirements include integrity checks (null coverage thresholds, duplication ratios, and outlier detection using configurable z-score bounds), end-to-end trace IDs across ingestion, reporting jobs, and exports, plus operational metrics for job duration, queue depth, and DB resource usage. Automatic retry uses exponential backoff capped at 3 attempts, and rollback is enforced via transactional boundaries for imports and itinerary edits; the entire system must run on a single offline Docker host with no external dependencies, with audit logs immutable for 1 year and sensitive fields masked in logs by default.
